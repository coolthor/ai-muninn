## 白話版：3 萬塊的桌機怎麼幫你看完英文影片

DGX Spark 是 NVIDIA 賣的小型 AI 桌機，台幣 9 萬多。我之前都拿它跑文字 — chat、寫 code、隨便聊。但同一顆晶片只要載入多模態 model，也能處理影片跟音訊。NVIDIA 出了一顆 **Nemotron-3-Nano-Omni-30B-A3B**，把三樣東西包在一起：看影格的 vision encoder、聽聲音的 Parakeet 音訊 encoder、還有 30B-A3B 的 LLM 把兩邊接起來（NVIDIA model card 寫總參數 ~31B，每 token 啟用 3B 的 MoE）。

我想做的事很單純：丟一支英文 YouTube 影片、podcast 片段、研討會演講進去，model 看完聽完之後幫我寫心得。不是純逐字稿、不是純畫面描述，是真的把講的東西跟看到的東西融合理解。

這篇就是把這個流程跑通的紀錄。前兩個雷讓我噴掉一個晚上，第三個是看影片長度怎麼調 knob。實測數據放在每個段落。

---

## 前言

[Part 25](/zh-TW/blog/dgx-spark-nemotron-3-nano-w4a16-74-toks) 在拚 30B model 單流文字 throughput 衝到 74.75 tok/s。這篇的目標完全不一樣 — 多模態正確性，速度不是重點。當你卡在「model 到底有沒有聽懂在講什麼」這種問題上，throughput 就不是 headline 了。

這篇講的兩個雷是 config 層級的，不是架構問題。Model 沒壞、vLLM serving 也沒壞。但兩個 flag 放對放錯，差別就是「真的聽懂」跟「自信滿滿地腦補」 — 而且 NVIDIA model card 範例兩個都沒明寫。

## 19 秒短片戳出來的音訊雷

第一支拿來測的影片是大名鼎鼎的「Me at the zoo」 — Jawed Karim 2005 年 4 月 23 日上傳到 YouTube 的史上第一支影片，19 秒，公開、極短、英文清楚。

<YouTubeEmbed id="jNQXAC9IVRw" caption="Jawed Karim, 'Me at the zoo'（2005-04-23）— YouTube 史上第一支影片，19 秒。" />

Model 第一次的回答畫面寫得很對 — 年輕男生穿藍衫加紅灰外套、兩隻大象、稻草、圍欄都有。但逐字稿吐給我的是這個：

> "I'm here at the San Diego Zoo. And I'm going to show you some elephants. There are three of them. One is eating hay..."

幻覺。YouTube 英文字幕軌記的實際旁白是：

> "Alright, so here we are in front of the elephants. The cool thing about these guys is that they have really, really, really long trunks. And that's, that's cool. And that's pretty much all there is to say."

整段沒提到 San Diego Zoo、也沒有三隻大象。Model 根本沒在聽 — 它從畫面推一段聽起來合理的旁白出來。畫面對得太準，害我想了一下才意識到。

修法藏在 vLLM chat parser 裡，`chat_utils.py:910`：

```python
# Extract audio from video if use_audio_in_video is True
if (
    video_url
    and self._mm_processor_kwargs
    and self._mm_processor_kwargs.get("use_audio_in_video", False)
):
    audio = self._connector.fetch_audio(video_url) if video_url else None
    audio_placeholder = self._tracker.add("audio", (audio, uuid))
    self._add_placeholder("audio", audio_placeholder)
```

沒這個 flag 的話，vLLM 根本不會從影片抽音軌、也不會餵給 Parakeet。Model 只拿到畫面，被迫去猜講者在說什麼。

flag 必須放在 chat completion request 的**最外層**，不是 `video_url` 物件裡：

```json
{
  "model": "nemotron-omni",
  "mm_processor_kwargs": {"use_audio_in_video": true},
  "messages": [{
    "role": "user",
    "content": [
      {"type": "video_url", "video_url": {"url": "data:video/mp4;base64,..."}},
      {"type": "text", "text": "把音訊轉錄出來。再描述畫面。"}
    ]
  }]
}
```

flag 移到外層之後再跑一次：

> "Alright, so here we are, one of the uh elephants. Um, cool thing about these guys is that they have really, really, really long um trunks. And that's that's cool. And that's pretty much all there is to say."

意思有了 — 大象框架對、"really, really, really long trunks" 這個關鍵錨點對、結尾也對。ASR 有點 drift（"one of the uh" 跟原版 "in front of the" 不太一樣，多了幾個 "um"），但語意保留。15 秒 wall time、5,637 prompt tokens、689 completion tokens。

## 3 分鐘 Karpathy 演講：把規模拉長

flag 修對之後，下一個問題就是「這台桌機能吃多長的影片」。我抓了 Andrej Karpathy「Intro to Large Language Models」的前 3 分鐘 — 講者頭像、投影片密集、技術內容硬核。

<YouTubeEmbed id="zjkBMFhNj_g" caption="Andrej Karpathy, 'The busy person's intro to LLMs' — 測試前 3 分鐘。" />

Model 吐出來的內容（結構是我整理的、原輸出帶時間戳）：

> Andrej Karpathy 自我介紹，提到他之前講過一個 30 分鐘的 LLM 演講沒被錄起來，所以重錄一次給 YouTube 用。標題投影片寫「The busy person's intro to LLMs」。下一張投影片是 `llama-2-70b` 的目錄，裡面兩個檔案：`parameters`（140GB）跟 `run.c`（約 500 行 C）。他解釋 parameters 檔 140GB 是因為 70 億參數每個用 2 bytes 的 float16 存。run.c 沒任何 dependency — 編譯成 binary、配上 parameters 檔，MacBook 不用網路就能跑這個 LLM。

每個 fact 都對。`llama-2-70b`、140 GB、~500 行 `run.c`、MacBook 跑 model — 全部對應到 Karpathy 影片真的講的內容。輸出是「整理過」而不是逐字稿，但技術點全保留。

| 指標 | 數值 |
|---|---|
| 影片長度 | 180 秒（3 分鐘）|
| Wall time | 89 秒 |
| Prompt tokens | 53,842 |
| Completion tokens | 2,000（撞到 max_tokens 上限）|
| 換算速率 | 每秒影片約 300 prompt tokens |

每秒影片 300 prompt token 這個數字兩支影片都吻合，是後面規劃長影片的基準。

## 另一個雷：別用 b12x image

[Part 25](/zh-TW/blog/dgx-spark-nemotron-3-nano-w4a16-74-toks) 我自己 build 了一個 `vllm-node-tf6-b12x` — vLLM 0.20.1 cherry-pick [PR #40082](https://github.com/vllm-project/vllm/pull/40082) 加 4 層 SM121 patch。那個 image 跑文字衝到 74.75 tok/s。第一直覺當然是 Omni 也用同一個 image。

跑不起來。第一個 decode step 就 NaN logits、output 全是 `!!!!!!!!!!!!`、engine 也救不回來。vLLM 啟動 log 寫的 dispatch 是這個：

```
Selected FlashInferFP8ScaledMMLinearKernel for ModelOptFp8LinearMethod
```

Omni 用的是 `modelopt_mixed` quantization — vision 跟 audio encoder 一個精度、LLM 主體另一個。我們那 4 層 SM121 patch 改了 FP8 GEMM kernel selection 的邏輯，剛好對這個混合 config 是錯的。我沒去 trace 到底是哪一行出問題，因為實務上的修法簡單很多：換 upstream image。

```bash
docker run -d --name vllm_upstream --gpus all --network host --ipc=host \
  --shm-size 16g \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  --entrypoint /bin/bash \
  vllm/vllm-openai:v0.20.0 \
  /tmp/launch-upstream-omni.sh
```

upstream `vllm/vllm-openai:v0.20.0` 沒我們的 patch、走的是 Omni 出廠原本測過的 kernel selection 路徑。同一台機器、同一顆 model、兩個 image 解兩個工作。

這不是「永遠分開」。意思是：當我們的 patched image 對某個 model 是錯的，那個 model 就獨立跑一個 stock-image container。硬要一個 image 包山包海，就是常見的整合錯誤。

## Mamba block_size assertion：啟動時的小坑

Upstream image 第一次起 `--max-model-len 65536` 在 engine init 就掛：

```
AssertionError: In Mamba cache align mode,
block_size (2128) must be <= max_num_batched_tokens (2048).
```

Nemotron-3 是 hybrid 架構、有 Mamba/SSM layer。Mamba 的 KV cache block size 是從 model config 算出來的，vLLM 預設 `max_num_batched_tokens=2048` 比那個 block size 小，所以撞到。修一個 flag：

```bash
--max-num-batched-tokens 8192
```

把 prefill 的 batch budget 調大就過了。我們單流 concurrency=1 用不到那麼多 — 這個 flag 是 prefill 一次 batch 多少 token 的**上限**，單支影片 request 根本不會碰到。

## 影片可以多長？

從 3 分鐘 Karpathy 那次的數據可以推：fps=2 下每秒影片約 300 prompt tokens。瓶頸是 `--max-model-len`、`num_frames`、跟採樣的 fps。

| 影片長度 | fps | num_frames 上限 | max-model-len | 預估 prompt tokens | 備註 |
|---|---|---|---|---|---|
| 1 分鐘 | 2 | 128 | 32768 | ~18k | 預設可以跑 |
| 2 分鐘 | 2 | 256 | 32768 | ~36k | max-model-len 拉到 65536 |
| **3 分鐘** | **2** | **512** | **65536** | **~54k** | **這篇實測的** |
| 5 分鐘 | 1 | 300 | 65536 | ~45k | 視覺細節砍一半，音訊不變 |
| 10 分鐘 | 0.5 | 300 | 98304 | ~45k | 視覺細節砍 3/4，手勢/投影片會漏 |
| >10 分鐘 | — | — | — | — | ffmpeg 切段、每段獨立處理、再串心得 |

這張表藏兩個取捨：

**fps↓ 省 token 但不影響音訊。**Parakeet 跑的是 16 kHz 音訊波形，跟 video frame rate 完全獨立。音訊 path 不管 fps 多少都產生大概每秒 12.5 個 audio token。所以降 fps 砍的是視覺 token 數跟視覺細節（手勢、投影片切換），音訊轉錄品質不變。長影片真正的勝場其實在這 — fps 降到 0.5、音訊一樣清楚。

**num_frames↑ + max-model-len↑ 吃統一記憶體。**GB10 的 128 GB 是 weight 跟 KV cache 共用。`max-model-len` 翻倍，KV cache budget 也翻倍。98k context 的時候，光一支影片 request 就吃掉桌機一半以上的 RAM。

超過 10 分鐘我會 ffmpeg 切段、每段跑一次、再用一個小型純文字 LLM 把 chunk 心得串起來。Single-pass 長影片不是這台 9 萬桌機的勝場。

## 可用的 recipe

`spark-vllm-docker/recipes/nemotron-omni.yaml`：

```yaml
recipe_version: "1"
name: nemotron-omni
description: Nemotron-3-Nano-Omni multimodal (video+audio) on upstream vLLM 0.20.0

model: nvidia/Nemotron-3-Nano-Omni-30B-A3B-Reasoning-NVFP4
container: vllm/vllm-openai:v0.20.0
solo_only: true

defaults:
  port: 8004
  host: 0.0.0.0
  gpu_memory_utilization: 0.80
  max_model_len: 65536
  max_num_seqs: 8
  max_num_batched_tokens: 8192

command: |
  pip install "vllm[audio]" --quiet && \
  vllm serve nvidia/Nemotron-3-Nano-Omni-30B-A3B-Reasoning-NVFP4 \
    --served-model-name nemotron-omni \
    --trust-remote-code \
    --gpu-memory-utilization {gpu_memory_utilization} \
    --max-model-len {max_model_len} \
    --max-num-seqs {max_num_seqs} \
    --max-num-batched-tokens {max_num_batched_tokens} \
    --kv-cache-dtype fp8 \
    --reasoning-parser nemotron_v3 \
    --enable-auto-tool-choice \
    --tool-call-parser qwen3_coder \
    --limit-mm-per-prompt '{{"video":1,"image":1,"audio":1}}' \
    --media-io-kwargs '{{"video":{{"fps":2,"num_frames":512}}}}' \
    --enable-prefix-caching \
    --host {host} \
    --port {port}
```

把 3 分鐘 MP4 丟進 `/tmp/`、base64 編完、request 最外層帶 `mm_processor_kwargs: {"use_audio_in_video": true}`，90 秒拿到一份「轉錄+畫面描述」的回答。

## 收穫

### 可以搬走的方法

**1. 多模態 model 在幻覺，先懷疑某個模態被 silent drop。**畫面對得很準、音訊內容卻是「合理但講者根本沒說過」 — 這個訊號就是某個 flag 本來該開但沒開。Grep 框架原始碼找那個 flag，確認 gating 條件真的有 truthy。

**2. 一個 image 不會適合所有 model。**為某個 model 做的 patch 可能打破另一個 model 的 dispatch。客製 image 上某個 model 跑壞，先 A/B 對照 upstream image，半數情況下 upstream 直接就跑得起來。

**3. 多模態的 token budget 跟著影片長度走，不是文字長度。**Nemotron-3-Nano-Omni 在 fps=2 是每秒影片 ~300 prompt tokens。前後文字 prompt 算不到主體 — 主體是 frame embedding。

### 通用原則

這台桌機一支多模態 request 3 分鐘影片就吃 5 萬+ prompt token，是純文字 chat 的 5 倍。KV cache 跟 prefill batch budget 在純文字場景平常根本不用想，到了影片場景變成現役限制。如果你打算在這台機器上跑多租戶 serving，影片 tenant 跟文字 tenant 必須分開排隊 — 兩個根本不在同一個 throughput 等級。

## 收尾

DGX Spark 真的能看英文影片告訴你內容。流程：

1. 用 upstream `vllm/vllm-openai:v0.20.0`，不要用 patched image。
2. 加 `--max-num-batched-tokens 8192` 過 Mamba block_size assertion。
3. `--max-model-len 65536`、`num_frames=512`、`fps=2` 撐到 ~4 分鐘。
4. 一定要把 `mm_processor_kwargs: {"use_audio_in_video": true}` 放在 chat request 最外層 — 沒這個 flag 音訊整個被 silent drop。
5. 影片 >5 分鐘就降 fps — Parakeet 跑 16 kHz 跟 fps 無關，音訊品質不會掉；>10 分鐘就 ffmpeg 切段。
